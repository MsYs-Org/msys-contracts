# MSYS i18n catalog v1

MSYS internationalization is a static resource contract, not a privileged
service. An application or role provider reads a UTF-8 JSON file from its own
package and formats the selected string in-process. This keeps the same catalog
usable from Python, C/C++, Qt, Electron, Tk, or another runtime without D-Bus,
gettext, ICU, a package manager, or a network dependency.

The normative structural schema is
[`msys.i18n.catalog.v1`](../schemas/i18n-catalog.v1.schema.json). Semantic rules
in this document that JSON Schema cannot express are normative too.

## Catalog shape

```json
{
  "$schema": "https://msys.local/schemas/i18n-catalog.v1.json",
  "schema": "msys.i18n.catalog.v1",
  "id": "org.example.clock",
  "default_locale": "en-US",
  "messages": {
    "en-US": {
      "window.title": "Clock",
      "alarm.summary": "Alarm at {time}"
    },
    "zh-CN": {
      "window.title": "时钟",
      "alarm.summary": "闹钟时间：{time}"
    }
  }
}
```

`id` scopes keys and normally equals the owning package id. Message keys are
stable identifiers, never source-language prose. Renaming a visible label does
not rename its key.

`default_locale` MUST name an entry in `messages`. Its message map is the
catalog's complete key set. Every key in another locale MUST exist in the
default locale; another locale MAY omit a key and inherit its default
translation. This makes partial translations safe without ambiguous cross-file
merge order.

Catalog files MUST be UTF-8 JSON without a byte-order mark, duplicate object
members, `NaN`, or infinities. Install-time validation SHOULD reject violations
before activation. Top-level experimental metadata is allowed only under an
`x-` name. Message maps deliberately contain strings only.

## Locale selection and fallback

Portable catalog locale names use a bounded BCP-47-style subset, for example
`zh-CN`, `zh-Hans-CN`, `en-US`, or `es-419`. Catalog names are canonical:
language is lower case, a four-letter script is Title Case, a two-letter region
is upper case, and variants are lower case.

An implementation selects its requested locale in this order:

1. an explicit application/API value;
2. `MSYS_LOCALE`;
3. `LC_ALL`;
4. `LC_MESSAGES`;
5. `LANG`;
6. the catalog's `default_locale`.

The environment is sampled when the translator is created or explicitly
reconfigured; no global locale mutation is required. POSIX spellings are
normalized before lookup: `zh_CN.UTF-8` becomes `zh-CN`; an `@modifier` and
encoding suffix are removed. `C`, `C.UTF-8`, and `POSIX` intentionally select
the catalog default.

For each lookup, candidates are deterministic. Start with the normalized
requested locale, remove one rightmost subtag at a time, keep candidates that
exist in `messages`, then append `default_locale` if it is not already present.
For example, `zh-Hans-CN` tries `zh-Hans-CN`, `zh-Hans`, `zh`, then the default,
but only for maps actually present. No title, country heuristic, system locale
database, or provider broadcast participates.

When one translation is intended for a whole language family, put the complete
translation in the language map (for example `zh`) and add `zh-CN` or
`zh-Hans` only for strings that genuinely differ. This makes `zh_CN.UTF-8`,
`zh-Hans-CN`, and a plain `zh` request converge on the same base without
duplicating a catalog in memory. Omit an exact-locale map when it has no
overrides; catalog message maps are non-empty by design.

If a key is absent from every candidate, a UI-facing implementation MUST return
the caller's literal fallback when supplied, otherwise the key itself. It MUST
make a `missing-key` diagnostic available and MUST NOT terminate the
application. Diagnostics are local data/callbacks, not broadcasts.

## Named placeholder grammar

Messages support only a small, deterministic grammar:

- `{name}` inserts the argument named `name`;
- names match `[A-Za-z_][A-Za-z0-9_]*`;
- `{{` emits a literal `{`, and `}}` emits a literal `}`;
- conversions, format specifications, indexing, attribute access, expressions,
  and nested placeholders are invalid.

The placeholder-name set for a translated key MUST exactly equal the set in
the default locale. Order and repetition MAY differ. Catalog validation rejects
unbalanced braces, unsupported syntax, unknown translated keys, and placeholder
set drift.

At render time an argument is either a Unicode string or a non-boolean integer
in the interoperable range `-9007199254740991..9007199254740991`. Strings are
inserted verbatim and integers use ungrouped base-10 ASCII. The bound is the
largest range represented exactly by Electron/JavaScript as well as native and
Python consumers. Dates, floating-point values, units, larger integers, and
locale-sensitive numbers must be formatted by the application first and passed
as strings. Argument text is data and is never parsed again.

A missing or invalid argument leaves its `{name}` token visible and emits a
diagnostic; extra arguments are ignored. A consumer MUST NOT implement this
contract with `eval`, expression interpolation, a shell, or unrestricted
language-native format syntax.

This intentionally omits ICU plural/select rules. Small-device applications
should use explicit keys such as `items.none`, `items.one`, and `items.many`
when wording differs. A future incompatible message grammar would use a new
schema identifier rather than changing v1 behavior.

## Package layout and ownership

The recommended package-relative location is:

```text
files/share/i18n/catalog.json
```

It is an ordinary packaged file; no manifest authority, role, permission, or
lifecycle semantic is implied. Components in the same package may share it or
ship separate catalogs with distinct ids. A package update replaces its own
catalog atomically with the rest of that immutable package version.

The active profile may inject `MSYS_LOCALE` into the clean component
environment. An application setting may instead pass an explicit locale to its
translator. Either approach uses the same fallback algorithm and requires no
long-running locale broker.

Launchers may localize immutable presentation metadata with the optional,
strict `x-msys-i18n` manifest extension. It does not change activation or grant
authority:

```json
"x-msys-i18n": {
  "catalog": "files/share/i18n/catalog.json",
  "name_key": "app.name",
  "summary_key": "app.summary"
}
```

`catalog` is a safe package-relative path. A component declaration overrides
the package declaration; missing or invalid metadata falls back to the normal
manifest `name` and `summary`. Consumers must keep the resolved catalog inside
the immutable package root and apply the same locale fallback rules as an app.
This avoids a global translation registry while allowing launchers and Settings
to present third-party applications in the selected language.

## Direct consumption from other languages

All consumers perform the same four operations: strict JSON load, schema/id
check, locale candidate construction, message lookup, and single-pass named
replacement.

- Python applications use the `msys_sdk.i18n.Translator` reference API.
- Qt/C++ applications may load the file with `QFile` and `QJsonDocument`, keep
  `messages` as `QJsonObject`s, and render with a small single-pass brace parser.
  This is separate from Qt's compile-time `.qm` mechanism.
- Electron/Node applications may use `fs.readFileSync(path, "utf8")` plus
  `JSON.parse`, then the same lookup and replacement rules. The package
  validator must already have rejected duplicate members, which `JSON.parse`
  alone cannot report.
- C applications may use the JSON parser they already bundle or convert a
  validated catalog to static tables at build time. MSYS does not require a
  target-wide JSON library and does not add one to the mIPC C ABI.

Consumers must not merge arbitrary catalogs by key. Select the catalog by its
stable `id`, then resolve a key inside it. This prevents install order from
silently changing UI text.
